# Going live on Arcium — integration plan & the architecture decision it forces

This document is the bridge from "tested locally" to "confidential on devnet." It covers how the
`shadowstate_mxe/encrypted-ixs/` circuits wire to Solana, the **hard conflict with our No-Anchor law**
and the recommended resolution, and the exact CLI/devnet workflow. The hands-on runbook is in
[`DEPLOYMENT.md`](DEPLOYMENT.md).

> Reference: the Arcium dev toolchain (Arcium **0.10.4**, Cerberus MPC, validated on devnet). All API
> specifics below track that toolchain.

---

## 1. How Arcium actually executes a computation (the three-instruction pattern)

Every MPC operation on Arcium is **three on-chain instructions**, all generated by `arcium-anchor`
macros on a Solana program:

1. **`init_*_comp_def`** — one-time: register the circuit (its off-chain `.arcis` URL + `circuit_hash!`).
2. **`<your_ix>`** — queue a computation: encrypt args with `ArgBuilder`, call `queue_computation(...)`
   with a chosen `computation_offset` and the callback descriptor. The ARX cluster picks it up.
3. **`<your_ix>_callback`** — the cluster, after running the circuit over secret shares, CPIs back
   into your program; `output.verify_output(&cluster_account, &computation_account)` authenticates
   the MPC result. You then write the (encrypted or revealed) output into your accounts.

Clients encrypt inputs with **x25519 + the Rescue cipher** (`@arcium-hq/client`) to the cluster key —
exactly what `mpc-core::seal` models. The encrypted batch book lives in an on-chain account as
`[[u8;32]; N]` ciphertext + a `nonce`.

Mapping to ShadowState:

| ShadowState step | Arcium instruction | Circuit |
|---|---|---|
| Open an epoch's book | `init_book` queue + callback | `init_book` |
| Client submits a sealed order | `ingest_order` queue + callback | `ingest_order` |
| FBA window closes (~1.2 s) | `clear_batch` queue + callback | `clear_batch` |

On the `clear_batch` callback the program holds the **revealed** clearing (`total_yes/total_no/
matched/net_imbalance/direction` + per-slot `(side, qty)`), which maps directly onto
`protocol::BatchHeader` + the per-fill split.

## 2. The conflict: real Arcium is Anchor-coupled

**This is the decision the "real Arcium" choice forces.** Arcium's entire on-chain SDK is Anchor:
`#[arcium_program]`, `arcium-anchor`, `arcium-macros`, the `queue_computation` / `verify_output`
machinery, and even the CLI (`arcium build` wraps `anchor build`). Our root law says **No Anchor,
absolute.** These cannot both be fully satisfied. The options:

| Option | What it means | Cost | No-Anchor stance |
|---|---|---|---|
| **A. Two-program split (recommended)** | A thin **Anchor `shadowstate-arcium-gateway`** owns *only* the Arcium plumbing (book account + the 3-instruction pattern). Our **Pinocchio `program/` stays the settlement engine** and is unchanged in spirit. | Low — write the gateway once. | Bends, doesn't break: the value-bearing settlement is still Anchor-free; only the unavoidable MXE glue is Anchor. |
| **B. Native Pinocchio Arcium client** | Reimplement `arcium-anchor` + `arcium-client` (comp-def PDAs, mempool/execpool, `ArgBuilder` serialization, `verify_output` cluster-attestation) natively in Pinocchio. | Very high + fragile; tracks Arcium internals across versions. | Honors the law literally. |
| **C. Drop Arcium** | Run our own MPC committee (the original `mpc-core` shape). | We operate a real MPC network. | N/A. |

**Recommendation: Option A.** It is how every shipped Arcium project is structured, keeps our
audited Pinocchio settlement engine intact, and isolates the Anchor dependency to a single small
program that does nothing but MXE orchestration. **This needs your explicit sign-off** because it
introduces Anchor into the repo — see the open decision at the bottom.

### A consequence to note: the committee verification changes role

With **real Arcium**, the trust anchor for "the batch result is honest" becomes Arcium's own callback
attestation (`verify_output` against the `cluster_account`), **not** our standalone Ed25519 committee.
So in Option A the settlement trust path is:

```
ARX cluster → clear_batch callback (verify_output) in the Anchor gateway
            → CPI into Pinocchio settlement, which trusts the gateway (caller/authority check)
```

Our native Ed25519 `Committee` (in `program/src/sig.rs`) then becomes the **non-Arcium fallback**
(Option C) rather than the primary path. The `program/` settlement instruction would gain a
"trusted-gateway authority" mode alongside the existing committee mode. This is an additive change,
not a rewrite.

## 3. Repo layout (as built)

> **Note (May 2026):** the early root-level `encrypted-ixs/` and `arcium-gateway/` 0.6.3 drafts have
> been **deleted**. The real Arcium **0.10.4** project lives in `shadowstate_mxe/` (created via
> `arcium init`, built with `arcium build`, gateway deployed to devnet `E3GF…dNXe`). In 0.10.4
> circuits are uploaded on-chain (no off-chain `circuits-repo` needed).

```
shadowstate_mxe/               ← THE Arcium project (built + deployed):
  ├─ encrypted-ixs/            ←   Arcis circuits (init_book / ingest_order / clear_batch)
  └─ programs/shadowstate_mxe/ ←   Anchor MXE gateway (book account + 3-instruction pattern)
program/                       ← Pinocchio settlement; trusted-gateway authority mode added
mpc-core/                      ← relayer (consume BatchCleared → pro-rata → SubmitBatchTrusted) + MPC model
mm-gateway/                    ← unchanged
client/                        ← TypeScript: seal orders → gateway ingest_order
```

`mpc-core` stops being the "MPC" (Arcium is now the MPC) and becomes the **client + relayer**: it
seals orders, queues `ingest_order`/`clear_batch`, and — on the revealed clearing — runs the existing
deterministic two-tier allocator (`mxe::allocate_exact`) to build the `protocol` frame for settlement.
Its sealing/matching/relay code is directly reusable.

## 4. Devnet workflow (Arcium toolchain)

Everything runs inside the Arcium Docker toolchain (Ubuntu 24.04 + Rust 1.89 + Solana 2.3.0 + Anchor
0.32.1 + `arcium` 0.10.4). See [`DEPLOYMENT.md`](DEPLOYMENT.md) for the container setup.

```bash
# 1. Compile circuits + gateway program + IDL
arcium build

# 2. In 0.10.4 circuits are uploaded ON-CHAIN (no off-chain repo) — done once via uploadCircuit.
#    This burst is what a free RPC rate-limits; use a paid endpoint or the Arcium localnet.

# 3. Deploy program + initialize the MXE on devnet (cluster offset 456)
arcium deploy \
  --cluster-offset 456 --recovery-set-size 4 \
  --keypair-path /root/.config/solana/id.json --rpc-url devnet \
  --program-keypair target/deploy/shadowstate_arcium_gateway-keypair.json \
  --program-name shadowstate_arcium_gateway

# 4. Init each comp def once (init_book / ingest_order / clear_batch), then run the loop
arcium test --cluster devnet --skip-build

# Diagnostics
arcium mempool 456 --rpc-url devnet
arcium computation 456 <computation_offset>
```

**Credentials reality check:** no secret login. You need (a) the `arcium` CLI + Docker toolchain,
(b) a funded Solana devnet keypair (computations are metered), (c) the devnet cluster offset (456),
(d) a public repo to host `.arcis`. That's the entirety of "onboarding."

Costs/limits to respect: division is ~3B+ ACUs (kept out of circuit); each computation needs a fresh
random `computation_offset` (new offset per retry); callback accounts must pre-exist and be sized;
`Shared` args always carry both pubkey **and** nonce in `ArgBuilder`.
